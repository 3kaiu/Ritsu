# 设计交付物模板 (Handoff Template)

> 当执行 Step 3 (方案设计) 完成后，AI 必须严格按照此模板输出最终设计方案。该文档必须达到“另一位 Agent 能够直接照着它写代码，而不需要重做决策”的标准。

## 1. 范围与上下文
- **正在构建的内容**：(1段话总结)
- **非目标（Out of Scope）**：(明确列出不在本次修改范围内的点)
- **当前依赖的配置文件/环境**：(如已读取到的 `package.json` 版本，禁止编造)

## 2. 方案详解
- **被选中的方案**：(为什么选它)
- **落选的备选方案**：(如果 tradeoff 差距 < 40%，简要记录为什么没选)
- **改动文件清单**：(预估修改哪些文件，具体到模块)

## 3. 架构攻击测试结果
*(方案必须经历以下三个维度的极端压力验证才能通过)*
- **依赖宕机生存力**：(如果外部接口/组件无响应，这套代码会怎样？)
- **激增 10x 验证**：(数据量或并发突然放大 10 倍，哪个函数最先崩？)
- **可逆性/回滚成本**：(如果上线后发现方向错了，回滚需要动数据库吗？)

## 4. 接口与契约 (如有)
- **对外暴露的 API / Props 变更**：(必须明确向下兼容策略)

## 5. 未决事项追踪 (Unknowns)
- (禁止使用模糊的 TODO。如果存在未知事项，必须指明：**“该问题延后决定的确切原因是什么”** 以及 **“下一步由谁/何种操作来解决”**)
