# 议题分诊与回复话术 (Issue Triage Rules)

> 触发时机：当用户提到“处理 Issue”、“看看 PR”、“Triage”时使用。

## 1. 核心处理原则 (Action-First)
- **事实优先**：不要使用分析性段落。直接检查状态：
  - 如果已经被 `git log` 中的提交修复 → 告知版本，关闭。
  - 如果被标记为 Duplicate → 指向原 Issue，关闭。
  - 如果是 PR → 倾向于帮贡献者修复（maintainer edit），而不是直接拒绝。
- **只说一次感谢**：最多在开头说一句感谢反馈，**禁止**在结尾添加任何冗长的寒暄（如“再次感谢您的支持，有其他问题随时联系”）。

## 2. 公共回复标准格式 (Public Reply Shape)
回复 Issue/PR 必须极度精简，像一个经验丰富的维护者：

1. **At 作者**：`@<login>` 开头。
2. **感谢（可选一次）**：`感谢反馈` / `Thanks for the report`.
3. **事实与原因（1-2句）**：客观描述导致问题的原因，或拒绝 PR 的具体技术阻碍。
4. **下一步行动（Next Step）**：
   - 必须提供可执行的下一步。例如：“在下一个版本 (v1.2) 中发布”、“请执行 `rm -rf cache` 再次重试”、“请补全复现的 Node 版本”。
5. **语言匹配**：如果提问者用中文/英文，必须用对应的语言回复。如果不属于这两种，统一使用英文回复。

### 错误范例 ❌
> "Hi @user, 非常感谢您花时间提交这个 Issue。我们分析了您的问题，确实是一个 Bug。我已经提交了修复，这将在下个版本发布。如果您还有任何问题，请随时在这里留言，再次感谢您的贡献！"

### 正确范例 ✅
> "@user 感谢反馈。这是因为缓存键值未能正确失效导致的。已修复，跟随下一版发布生效。遇到急用可先运行 `npm run clean:cache` 绕过。"
