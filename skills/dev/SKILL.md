# Dev: 领域严苛的纯净编码 (Adaptive Implementation)

**触发条件**：用户输入 `/r-dev`。

## 核心职责 (Capability Convergence)

把你接到的方案转化为极其健壮的生产代码。**强制收束发散思维，禁止越权修改需求范围**。

## 执行流水线

### 1. 领域纪律确认

- 展示已加载的领域标识及对应 `domains/[domain].md` 中的编码纪律摘要。
- 确认当前编码将严格遵守这些纪律。

### 2. 领域专属编码纪律执行 (Domain-Specific Strictness)

**禁止写面条代码，按照你的领域严格执行纪律：**

- 执行 `domains/_base.md` 中的通用编码纪律。
- 执行 `domains/[domain].md` 中的领域增量编码纪律。

### 3. 未知标识符绝对拦截 (Unknown Identifier Check)

这是 Ritsu 的最高安全红线！

- 在你写下并提交任何代码，尝试调用一个外部模块的函数或变量前，**必须先使用 `grep` 工具验证它在项目中真实存在**。
- **类型签名验证**：不仅验证存在性，还必须验证调用参数与函数签名的匹配度（参数数量、类型、必选/可选）。若无法静态验证，必须在验证步骤中运行类型检查。
- 绝不允许凭记忆捏造一个系统里根本没有的方法名。一旦导致编译报错，视为严重失职。

### 4. 测试先行 / 验证前置 (Test-First)

写业务代码前，必须先确立"如何证明我写对了？"

- **策略分级**：
  - **标准路径**（新功能/重构）：必须先写出对应的单测用例，TDD 红灯→绿灯。
  - **快速路径**（紧急 hotfix）：至少提供一个验证用的 `curl` 命令或清晰的 UI 验证步骤，修复后补单测。
  - **契约驱动路径**（有 HANDOFF.md）：验收标准直接取自 Handoff 的验收条件，逐条验证。

### 5. 沙盒式自查与零占位符 (Zero-Tolerance)

写完不要立即宣称完成，先在后台做一次假想的 `git diff` 级别的比对：

- 遗留了孤儿引用？产生了未使用的残余变量？必须立刻清理。
- **零占位符红线**：如果代码里发现 `TODO`, `//后续完善` 等推卸责任的字眼，必须自我推翻重写。
  - **唯一例外**：等待上游 API 确认的临时标记，必须以 `// BLOCKED: [具体阻塞原因 + 预计解除时间]` 格式标注，且在交付清单中显式列出。

### 6. 强制项目校验 (Validation)

必须读取 `AGENTS.md` 中规定的 Lint 和测试命令并在终端执行。完全跑绿才能交付。

**交付模板**：

```markdown
# 律 (Ritsu) 开发落盘清单

- **涉及文件**：(+改动点概述)
- **变更影响范围**：(是否影响 API 接口 / 数据库 Schema / 其他模块调用方式)
- **校验结果**：(Lint与验证结果，附执行命令)
- **验收条件达成**：(逐条对照 Handoff 验收标准，✅/❌)

> 技能流转参见 `state-machine.md`。

- **BLOCKED 项**：(如有，列出阻塞原因及预计解除时间)
```
