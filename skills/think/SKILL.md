# Think: 需求分析与设计单产出

**触发条件**：用户输入 `/r-think`。

## 执行流水线

### 1. 现状加载与对账

> 引用 `_shared/skill-common-steps.md` Step 0

读取上下文，重点检查：
- 是否有未完成的任务
- 最近的交付记录（如果是回流任务）

### 2. 技术栈感知与倾向性识别 (Stack Perception)

在深入分析前，必须通过 `read-agents` 和文件系统扫描识别当前项目的**倾向性**：
- **子领域识别**：它是 Frontend-React、Frontend-Flutter、Backend-Node 还是其他？
- **架构倾向**：项目是否偏向特定的状态管理（如 Redux vs Bloc）、请求层或样式系统？
- **人格设定**：AI 必须明确宣告本次任务的“资深架构师”人设（例如：“我将以 Flutter 资深架构师身份进行性能优化”）。

### 3. 深度架构与需求审核

从**资深架构师**视角分析目标、边界与风险：
- **目标与核心链路**：这次到底要做成什么样？对现有核心业务链路的影响是什么？
- **边界与非功能需求**：明确什么是不做的。考虑性能（LCP/FID/CLS）、可靠性与安全性。
- **架构侵入度**：识别修改是否破坏了现有的设计模式或引入了技术债。
- **依赖分析**：必须使用 `ritsu_query_kg` 检查受影响的反向依赖闭包。

### 3. 设计单 (Design Sheet) 产出

将分析结果落盘为 **`design-sheet`**，强制包含以下架构维度：
- **方案范围**：受影响的文件及模块。
- **核心契约**：API 变更、组件 Props、状态流向。
- **性能优化清单**：算法复杂度优化、渲染频率控制、缓存策略。
- **实施清单**：原子化步骤。
- **验证计划**：不仅是功能测试，还需包含契约覆盖率与性能基准。

### 4. 交付摘要与引导

> 引用 `_shared/skill-common-steps.md` Step 4（skill=think）

**强制引导语**：
- 在输出摘要后，明确告知用户设计已完成。
- **示例**：“架构设计已就绪 (`design-sheet-xxx.md`)。本次方案优先采用了 `react-use` 优化副作用并精简了 API 请求层。如果你满意，请运行 `/r-dev` 开始实现。”
