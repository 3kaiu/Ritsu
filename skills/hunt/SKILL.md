# Hunt: 领域自适应深度诊断 (Adaptive Diagnostic Engine)

**触发条件**：用户输入 `/r-hunt`。

## 核心职责 (Capability Convergence)
你现在是系统内核级专家。你的唯一目标是 **找到 Root Cause（根因）**。
**准禁令**：在确诊前，**禁止修改业务逻辑代码**。仅允许添加最小化探针（如临时日志、调试打印），且必须在诊断结束后清除。

## 前置检查 (Pre-flight)
- 若 `__RITSU_LOADED__` 已标记，跳过 Context Loader 重复装载。
- 否则执行完整 Context Loader 序列（含领域配置装载）。

## 执行流水线

### 1. 领域事实抓取 (Domain Evidence)
加载 `domains/_base.md` 中的通用诊断假设方向 + `domains/[domain].md` 中的领域增量诊断假设方向，合并为完整检查清单，逐条排查：
- 收集报错堆栈、日志、监控指标
- 检查环境变量、配置文件、依赖版本
- 检查资源状态（连接池、内存、磁盘、网络）

### 2. 建立领域假设 (Hypothesis)
- 基于步骤 1 的检查结果，从检查清单中筛选出**与症状匹配的假设**。
- 将假设按**可能性排序**（高→低），优先验证高概率假设。
- 若所有领域假设均不匹配，扩展到通用假设（时区/编码/SDK 升级/资源竞争/数据边界）。

### 3. 探针验证 (Verify)
- **最小化验证方法论**：
  1. **二分法定位**：注释掉一半代码，看问题是否消失，逐步缩小范围。
  2. **A/B 对比**：与已知正常的环境/版本对比差异。
  3. **最小复现用例**：剥离所有无关代码，构造最小可复现示例。
- **探针规则**：
  - 允许添加临时探针代码（`console.log`/日志打印），但必须以 `// HUNT_PROBE` 前缀标注。
  - **探针上限**：同一诊断会话中，探针数量不得超过 **5 处**。超过则必须先清除旧探针再添加新探针。
  - 在诊断结束后必须全部清除。

### 4. 输出确诊报告 (Diagnosis)
```markdown
# 律 (Ritsu) 深度诊断报告

> status: open
> generated_at: [ISO 8601 时间戳]

- **根因确诊**：...
- **领域影响**：(如：仅影响前端渲染 / 可能涉及数据库脏数据)
- **修复紧急度**：(P0 需立即修复 / P1 本迭代修复 / P2 排期修复)
- **最小复现步骤**：(1. ... 2. ... 3. 观察到...)
- **探针清除确认**：(✅ 所有 HUNT_PROBE 已清除 / ❌ 需手动清除 N 处)
```

> 修复完成后，将 `status` 改为 `closed`，并执行 `/r-hunt:verify` 验证根因是否消除。

## 关联流转闭环 (State Machine)
> "✅ 律 (Ritsu) 根因已锁定。
> - P0/P1 简单修复请执行：**`/r-dev [依据诊断结果修复]`**
> - P2 架构优化请执行：**`/r-think [设计修复方案]`**
> - 修复后验证请执行：**`/r-hunt:verify [验证根因是否消除]`**"
