# Review: 领域自适应对抗审查 (Adaptive Adversarial Check)

**触发条件**：用户输入 `/r-review`。

## 核心职责 (Capability Convergence)
站在“拒绝合并”的对立面来审查代码。不仅查 Bug，更要执行 **溯源对账**：确保代码完美还原了 `/r-think` 的 Handoff 设计。

## 执行流水线

### 1. 抓取上下文与溯源对账 (Handoff Traceability)
- 自动运行 `git diff HEAD` 拿到当前变更。
- **强制对账**：如果存在 `/r-think` 产出的 Handoff 设计案或 PRD，必须先对比。
- **拦截红线**：若代码偏离了原定接口契约、漏掉了 PRD 上的状态展示、或擅自改变了架构层级，视为 **架构漂移 (Handoff Drift)**，直接打回。

### 2. Hard Stops (绝对红线拦截)
无论什么领域，命中以下红线之一，直接拒绝合并：
1. **不明标识符**：代码中引入了找不到定义的变量或组件。
2. **版本号不同步**：`package.json` 等多处版本号（如锁文件）出现割裂。
3. **明文凭证泄露**：存在硬编码的 Token、Key、或敏感日志全量打印。
4. **破坏性契约变更 (Breaking Changes)**：修改了现有的 API 结构或数据格式，但不向后兼容，且没有提供双写/迁移方案。

### 3. 领域动态对抗审查 (Domain-Adaptive Adversarial Pass)
根据代码领域（后端/前端/全栈）动态调整审查人格的逼问焦点：

- **若是【后端代码】审查焦点**：
  - **SQL 注入与越权**：查询语句是否安全？接口是否只需知道 ID 就能绕过鉴权横向越权？
  - **竞态与死锁**：高并发下这段事务是否会导致死锁？是否存在非原子的“先查后改”逻辑漏洞？
  - **异常兜底**：异常捕获后是否造成了数据库连接未释放或事务未回滚？

- **若是【前端代码】审查焦点**：
  - **XSS 与数据注入**：用户输入是否直接渲染在 DOM 中？
  - **状态篡改**：前端状态管理是否容易被直接从 Console 或插件中篡改并绕过业务校验？
  - **性能与渲染**：是否存在不必要的大面积重渲染？大列表是否做了虚拟化？
