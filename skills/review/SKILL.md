# Review: 多线化身与对抗审查 (Adversarial Check)

**触发条件**：用户输入 `/review` 或请求审查。

## 核心职责 (Capability Convergence)
主动拉取当前的 Git Diff，不留情面地进行安全与架构审查。如果是简单的代码修补，走快速审查；如果是核心逻辑或大于 500 行的变更，启动**对抗审查引擎**。

## 执行流水线

### 1. 抓取上下文
- 自动运行 `git diff HEAD` 拿到当前变更。如果 Diff 是空的，询问是否审查特定的提交。

### 2. 规模分级 (Scope-based Depth)
计算改动行数，决定审查烈度：
- **< 100 行**：快打快回。只查语法、逻辑漏洞、基础遗漏。
- **100 - 500 行**：防腐层审查。检查是否引入了循环依赖、公共库是否被污染。
- **> 500 行 或 涉及支付/鉴权/路由**：全量启动对抗审查（化身机制）。

### 3. 多线化身审查 (Adversarial Pass)
如果触发了对抗审查，你必须在后台扮演以下三个专家，并将三个视角的漏洞合并输出。

1. **架构师 (Architecture Lead)**
   - “这次改动破坏了单向数据流吗？”
   - “新增的底层 Service 是不是偷偷依赖了上层的 UI Controller？”
2. **安全守门员 (Security Guard)**
   - “这个输入如果是一个超长恶意字符串或 NULL，系统会崩吗？”
   - “日志里打印的 `res.data` 有没有把 Token 连带打印出来？”
3. **恶意的黑客 (Adversarial Attacker)**
   - “我要怎么利用这个新开的接口把系统的数据库打挂？”
   - “这块支付重试逻辑，我能不能通过高并发请求实现一次购买多次发货？”

**合并与收敛规则**：多个专家发现的相似漏洞合并为一个高危漏洞输出。无意义的“语法建议”全部过滤，只抛出“改变系统行为的漏洞”。

## 关联调用流转 (Sequential Invocation)
输出最终审查报告，给出具体修改点（如果有）。如果没有问题：

> "✅ 律 (Ritsu) 审查完毕，未发现致命缺陷。
> 你可以放心合并代码或执行发布流程了。"
