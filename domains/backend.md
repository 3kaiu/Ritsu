# Domain: Backend (后端)

> 继承 `_base.md` 全部内容。以下仅声明后端领域增量。

## 领域增量诊断假设方向
- 死锁 / 连接池耗尽 / OOM
- 事务隔离级别导致的不可重复读 / 幻读
- 非幂等写入导致的重复数据
- 上游微服务响应延迟 / 熔断触发

## 领域增量编码纪律
- **事务边界**：所有的多表/多写操作必须包裹在事务中。
- **日志规范**：重要链路必须有 Trace/Info 日志，严禁吞异常不打日志。

## 领域增量审查攻击向量
- **SQL 注入与越权**：查询语句是否安全？接口是否只需知道 ID 就能绕过鉴权横向越权？
- **竞态与死锁**：高并发下这段事务是否会导致死锁？是否存在非原子的"先查后改"逻辑漏洞？
- **数据迁移安全**：破坏性 Schema 变更的迁移脚本是否可逆？是否有回滚测试？

## 领域增量 PR 合并要求
- 必须提供 **单测覆盖率报告** 或 **关键路径的 Benchmark 性能报告**
- 检查是否涉及破坏性 Schema 变更

## 纪律-攻击向量对偶映射 (领域增量)
| 编码纪律 (防守) | 审查攻击向量 (测试防守) |
|---|---|
| 事务边界 | 竞态与死锁：事务是否原子？先查后改是否有漏洞？ |
| 日志规范 | 异常兜底：异常是否被吞导致无日志可查？ |
