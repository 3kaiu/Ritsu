# Domain: Base (通用基线)

> 所有领域配置隐式继承本基线。领域文件中不得重复声明以下内容。

## 领域文件接口契约

每个 `domains/[domain].md` 必须按以下顺序声明且仅声明增量内容（与 `_base.md` 重复的禁止出现）：

1. **领域增量诊断假设方向** — 该领域特有的 bug 假设清单
2. **领域增量编码纪律** — 该领域必须遵守的编码约束
3. **领域增量审查攻击向量** — 该领域特有的安全/质量攻击面
4. **领域增量 PR 合并要求** — 该领域 PR 必须满足的额外条件
5. **纪律-攻击向量对偶映射 (领域增量)** — 编码纪律与攻击向量的对应关系

## 通用诊断假设方向

- 时区 / 编码 / Locale / 序列化差异
- 第三方 SDK 静默升级或 API 变更
- 环境变量缺失或配置漂移
- 资源竞争（文件锁、端口占用、连接池耗尽）
- 数据边界（空值、超长文本、特殊字符、Unicode 异常）

## 通用编码纪律

- **资源释放**：所有外部资源（连接、句柄、流、会话）必须在 `finally` / `defer` / 组件卸毁中释放。
- **错误传播**：严禁吞异常。捕获后必须：重新抛出、记录日志、或转换为领域错误。空 catch 块视为严重违规。
- **幂等性**：写入操作应尽量设计为幂等。非幂等操作必须显式标注并说明原因。

## 通用审查攻击向量

- **输入校验断裂**：任何外部输入（用户输入、API 响应、文件内容）是否在入口处做了类型/长度/格式校验？
- **权限越界**：是否只需知道 ID 就能操作非归属资源？（IDOR 检查）
- **回滚能力**：变更是否可逆？回滚步骤是否文档化且测试过？

## 通用 PR 合并要求

- CI 必须全绿（或手动执行 `AGENTS.md` 质量门禁命令）
- PR 描述必须包含变更动机和影响范围
- 无 Hard Stop 命中（参照 `/r-review` 红线列表）

## 纪律-攻击向量对偶映射

| 编码纪律 (防守) | 审查攻击向量 (测试防守)                  |
| --------------- | ---------------------------------------- |
| 资源释放        | 异常兜底：捕获后资源是否未释放？         |
| 错误传播        | 输入校验断裂：异常是否被吞导致静默失败？ |
| 幂等性          | 重复请求是否产生脏数据？                 |
